# PR 생성 규칙

## base 브랜치 매핑

| 현재 브랜치 접두사 | base 브랜치 | 머지 후 추가 조치 |
|---|---|---|
| `feature/` | `develop` | 없음 |
| `bugfix/` | `develop` | 없음 |
| `release/` | `main` | 머지 후 `main` → `develop` 별도 머지 필요, 태그 생성 권장 |
| `hotfix/` | `main` | 머지 후 `main` → `develop` 별도 머지 필요, 패치 버전 태그 생성 권장 |
| `support/` | 해당 유지보수 브랜치 | 없음 |

## PR 제목 형식

```
[<TYPE>] <간결한 요약>
```

- `<TYPE>`: 브랜치 접두사를 대문자로 (`FEATURE`, `BUGFIX`, `RELEASE`, `HOTFIX`, `SUPPORT`)
- 이슈 번호가 있으면 뒤에 덧붙임: `[FEATURE] 소셜 로그인 기능 추가 (PROJ-101)`
- 커밋이 여러 개라도 PR 제목은 전체 작업을 대표하는 한 문장으로 요약

## PR 본문 템플릿

```markdown
## 개요
(이 PR에서 무엇을, 왜 했는지 2~3줄 요약)

## 변경 사항
- (커밋 그룹 1을 한 줄로)
- (커밋 그룹 2를 한 줄로)
- (커밋 그룹 3을 한 줄로)

## 테스트
- (어떻게 검증했는지: 유닛 테스트 추가, 수동 테스트 시나리오 등)

## 관련 이슈
Closes #<issue-number>  (또는 Refs: PROJ-123)

## 스크린샷 (UI 변경 시)
(해당 없으면 섹션 생략 가능)
```

## 본문 작성 규칙

- "변경 사항" 목록은 3단계에서 만든 커밋 그룹과 1:1로 대응시킵니다 (커밋 목록만 봐도 무엇을 했는지 알 수 있어야 함).
- 관련 이슈 번호가 대화나 브랜치 이름에서 확인되지 않으면 "관련 이슈" 섹션은 생략하고 사용자에게 알립니다. 임의로 이슈 번호를 만들어내지 않습니다.
- `release/`, `hotfix/` PR은 "테스트" 섹션에 회귀 테스트 여부를 반드시 포함합니다.

## gh CLI 호출 예시

```bash
gh pr create \
  --base develop \
  --head feature/social-login \
  --title "[FEATURE] 소셜 로그인 기능 추가 (PROJ-101)" \
  --body "$(cat <<'EOF'
## 개요
구글/카카오 소셜 로그인을 추가했습니다.

## 변경 사항
- feat(auth): 소셜 로그인 API 및 OAuth2 플로우 구현
- test(auth): 소셜 로그인 통합 테스트 추가
- docs(readme): 소셜 로그인 설정 방법 문서화

## 테스트
- 신규 유닛/통합 테스트 12건 추가, 전체 통과 확인
- 구글/카카오 실제 계정으로 수동 로그인 테스트

## 관련 이슈
Refs: PROJ-101
EOF
)"
```
